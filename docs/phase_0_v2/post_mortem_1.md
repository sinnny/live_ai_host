## 3. What failed and why

### 3.1 Root cause — sprite identity drift between independent generations

Each of the 10 expression sprites (neutral, cheek_stuff, panic, pleading, victory, sleepy, confused, sneaky, laughing, embarrassed) was generated by an **independent Qwen-Image inference** with the LoRA loaded. The LoRA constrains identity, but does not enforce frame-to-frame continuity that doesn't exist in its training signal. Result:

- Each sprite has slightly different lighting tone
- Headset and apron sit at slightly different positions / angles
- Head-to-body proportions drift by a few percent
- Color hue varies (~3-5% in fur tone)

When the renderer swaps between expression sprites at script boundaries (every 3-9 sec per segment), the swap reads as "**a different but similar Daramzzi just appeared**" — not as "**the same Daramzzi changed expression**". 300ms crossfade smooths the swap but cannot reconcile the underlying difference.

This is a **structural limitation**, not a tunable parameter. Tighter LoRA training, ControlNet conditioning, IP-Adapter anchoring, etc. can reduce but not eliminate it — the fundamental issue is that independently-generated frames do not constitute "the same character moving."

### 3.2 Root cause — amplitude lip-sync flicker

The Phase 0 simplified rig (Path A, committed in 9708711) uses an amplitude envelope from `phoneme_alignment.py` to toggle the expression layer between the segment's intended expression and `laughing` (an open-mouth expression). The amplitude detection emits ~50ms windows of `aa` vs `closed`.

For typical Korean speech with natural micro-pauses + breathing, this toggle fires every 100-300ms. With a 300ms expression crossfade, the crossfade rarely completes — the renderer is perpetually mid-transition. Visually this reads as:

- Flickering / shimmering during speech
- Character "vibrating" between two states
- Compounds with the sprite drift in §3.1 — every micro-flicker reveals more cross-sprite inconsistency

The flicker could be partially fixed with hysteresis (require sustained 200-500ms before swap) but the underlying problem — that expression swap is the only motion primitive — remains.

### 3.3 Root cause — overlay sprite generation fundamentally failed

The original FSD design assumed isolated mouth/tail/ears overlay sprites that composite over the expression layer with z-ordering. Two attempts to generate these failed:

**Attempt 1 (with LoRA loaded):** Every "isolated mouth on white background" prompt produced a full-character Daramzzi. LoRA's training distribution (9 full-character images) override the prompt's isolation instruction.

**Attempt 2 (LoRA disabled, base Qwen-Image only):** Even without LoRA bias, Qwen-Image misread "ISOLATED close-up of ONLY the mouth area" — produced full characters with different mouth shapes, or randomly cropped body regions (the `ears/up.png` came out as just the lower body with no head visible).

This failure forced the Path A simplified rig (expression-only), which in turn forced the amplitude lip-sync flicker problem above. **The overlay-sprite assumption in the FSD is not achievable with current OSS image-gen.**

### 3.4 Contributing factor — TTS quality

CosyVoice 2's `inference_cross_lingual` produces Korean with a Chinese-female timbre that founder reaction described as "androgynous" and "completely weird". `inference_instruct2` with a Chinese voice-style prompt caused output language to leak to Chinese. Segment 0 of clip_01.json synthesized as 21 seconds for what should be 3-4 seconds of speech (CosyVoice silence bloat at start of generation).

The TTS isn't the primary failure — even with perfect TTS, the visual rig would still fail #2 and #4. But the bad audio compounds the founder's negative reaction. **Real fix: native Korean TTS** (deferred — likely paid Korean provider when we revisit).

## 4. Path forward — three options

### Path B1 — EchoMimic v3 on stylized Daramzzi (Recommended for next attempt)

EchoMimic v3 failed on **photoreal human + Korean lip-sync** in Phase 0 v1 (per `memory/project_echomimic_v3_findings.md`). The hypothesis worth testing: does this failure translate to **stylized cartoon mascot** input?

**Argument for trying:**
- PRD §1.2 "obviously AI / failure-modes-as-style" thesis: imprecise lip-sync on a cartoon mouth reads as character quirk, not as broken. The exact failure that killed it on humans may be acceptable here.
- Cartoon faces likely more forgiving of geometric mismatch than realistic faces.
- Cheap to validate: existing Phase 0 v1 setup is preserved; just feed it the Daramzzi `seed.png` (or `expression/neutral.png` from the new atlas) + a clean Korean TTS WAV.
- If it produces smooth talking motion, it solves both sprite drift (single source image, model generates motion) and amplitude flicker (model handles lip motion natively).

**Risk:** EchoMimic v3 was trained on photoreal human faces. Stylized chibi proportions may produce uncanny artifacts. Unknown until tested.

**Cost:** ~1 day to integrate + 1 day to evaluate. ~$10 GPU.

### Path B2 — Live2D / Inochi2D rigged 2D character

Live2D (commercial) or Inochi2D (FOSS) take a single character art and let an artist build a mesh + rig that deforms in real time driven by parameters (mouth open/close, eye blink, head turn). Used heavily in VTuber commerce.

**Argument for:**
- Direct fit for the obviously-AI mascot use case.
- VTuber commerce on Naver 쇼핑라이브 already validated the format.
- Motion is the same character deforming, not different characters appearing — solves §3.1 entirely.

**Risk:** Requires manual rig work (the PRD §0 Pivot 2 explicitly rejected this for the "no humans, AI all the way down" reason). Reintroducing this contradicts the founder's stated thesis. Would need a rig artist OR a heroic effort to automate the rig generation.

**Cost:** ~1 week per character if rig automation works, or per-character manual rigging cost otherwise.

### Path B3 — Paid talking-head APIs (Hedra / Kling Avatar 2.0 / OmniHuman 1.5)

Per `memory/tech_stack_2026.md`: these are the SOTA 2026 talking-head APIs that take a single image + audio and produce realistic talking video. Founder previously flagged Kling as paid (negative signal). But these are the most reliable path to "good visual."

**Argument for:**
- Highest probability of producing a watchable result.
- Avoids the OSS quality gap that killed Phase 0 v1.
- Per-clip cost is on the order of $0.50–$2.

**Risk:** Per-stream cost in production would be significant. At PRD §5.2 target ₩30,000/broadcast (~$22), even $5/hr of talking-head API eats margin fast. Paid APIs also create vendor lock-in and license risk.

**Cost:** ~$1-3 per prototype clip to validate. Per-broadcast in production: TBD, needs Phase 1 pricing analysis.

## 5. Recommendation: Path B1 first, fall back to B3

**Sequence:**

1. **Next session: try EchoMimic v3 on Daramzzi.** Feed in `atlas/expression/neutral.png` (or `seed.png`) + a clean Korean TTS WAV. Measure: does it produce watchable motion at all on a chibi character?
2. **If B1 works** → Phase 0 v3 passes, proceed with EchoMimic v3 as the v1 production rig. The "obviously AI" thesis carries the lip-sync imprecision.
3. **If B1 fails** → escalate to B3. Test Hedra (image + audio → talking video, $0.30-1/clip). Hedra is specifically marketed for character animation.
4. **B2 (Live2D)** stays deferred. Reintroducing manual rig work contradicts the "AI all the way down" thesis; only revisit if both B1 and B3 fail.

**Decision pending founder approval.**

## 7. Lessons for next attempt

1. **Test motion quality with a 3-sprite atlas first** before committing to LoRA training a full atlas. We trained a 1.2 GB LoRA before discovering the sprite-swap motion is structurally choppy. Could have caught this with a hand-stubbed 3-image atlas in 30 minutes.

2. **OSS image-gen cannot reliably produce "isolated body part on white background".** Both LoRA and base model fail at this prompt regardless of negative-prompt engineering. Any future rig design that assumes isolated overlays is doomed. Either: (a) use full-frame sprites with built-in mouth states (combinatorial explosion), (b) use rigged 2D (Live2D), (c) use video-based motion generation (talking-head model).

3. **Korean TTS via Chinese-trained models has both timbre and tempo issues.** CosyVoice 2's cross-lingual mode loses voice tone consistency and adds silence bloat. For v1 production, a paid Korean TTS provider is likely required regardless of how the visual rig is solved.

4. **Renderer infrastructure (Playwright + three.js + ffmpeg) is rig-agnostic and reusable.** The orchestration code, TTS pipeline, phoneme alignment, encode stage all carry forward unchanged. Only `puppet.js` + `timeline.js` + `atlas_loader.js` need swapping if we move to a different visual approach.

5. **`make_mascot` pipeline carries forward.** The 8-stage atlas generator and Daramzzi LoRA are reusable. If we move to talking-head video, we feed it the atlas. If we move to Live2D, we feed an artist the seed. The character identity work is sunk-cost in a good way.

6. **Render speed is dominated by `page.screenshot(type='png')` at 1080p (~400ms/frame), not by Three.js render itself.** Today's render took 43 minutes for 5775 frames (~446 ms/frame). If we ever return to a frame-capture-based renderer (Phase 1 sprite-puppet retry or test_3 live capture), the cheapest wins are:
   - Switch `type='png'` → `type='jpeg', quality=85` (~2× faster, quality drop invisible for prototype evaluation)
   - Drop 60 fps → 30 fps (~2× faster — 24-30 fps is sufficient for lip-sync)
   - Render at 720p then upscale at encode (~2× faster)
   - **Live stream mode** (FSD §4.1): Playwright virtual webcam → ffmpeg pipe, bypass PNG entirely. ~5-10× faster but requires v4l2loopback setup and is the right answer for test_3 live runtime.
   - Multiple Playwright contexts/tabs rendering frame ranges in parallel.

   Stacking JPEG + 30 fps + 720p alone would have cut today's 43-min render to ~5 min. **Not relevant for the immediate next test (EchoMimic v3 produces video directly, no frame-by-frame), but is relevant for any future rig that returns to sprite-puppet or for the eventual live-runtime test_3 path.**
