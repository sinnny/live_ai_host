01

목표와 품질 기준

무엇을 만들려 했고, 어떤 기준으로 통과/실패를 판정하는가.

제품 정의

한국 라이브 커머스 시장을 대상으로, 방송 1회당 ≈ ₩30,000 단가로 제품 소개·시청자 응답·재미 요소를 갖춘 AI 라이브 호스트 SaaS.

1차 타겟 고객

한국 SMB 셀러 — 자체 라이브 운영 인력·인프라가 없고, 식품·생활용품 등 라이브 적합 카테고리를 다루는 중소 판매자.

품질 기준 (PRD §7.1)

  • 시청자가 "AI인지 사람인지 구분되지 않거나, 구분되더라도 시청 거부감이 없을 것"
  • 한국어 자연 발화 — 립싱크 · 억양 · 자연스러운 끊어 말하기
  • 실시간 댓글 반응 — 단순 자동응답이 아닌 살아 있는 상호작용

경제성 기준

SMB 셀러가 부담 가능한 1회 방송 단가 ₩30,000 (≈ $22 / hr). 이 단가를 못 맞추면 1차 타겟에 도달 불가, GTM 자체가 흔들립니다.

토킹헤드 실험 — 10개 영상

OSS 4개 + 유료 API 6개. 모든 단가는 RunPod 실측 시간당 × wall time, 또는 fal 청구액 그대로.

빠른 검증을 위해 실사 인물 사진을 reference로 사용. 프로덕션에서는 §3.1의 얼굴 생성 단계를 먼저 거쳐 AI 생성 페르소나를 만든 뒤 토킹헤드에 투입하는 순서가 됩니다.

4.1 오픈소스 (OSS) — 4개

두 가지 독립 OSS 접근을 검증: ⓐ EchoMimic v3 Flash 3개 config + ⓑ sprite-puppet rig 1개.

비용: $0.49/hr × 14.4m = $0.118
실패 한국어 자음 폐쇄·개방 입 모양 미반영.

스택: Wan2.1-Fun-1.3B + chinese-wav2vec2-base + EchoMimic v3 Flash. RunPod A6000 48GB.

② EchoMimic v3 · Config B

변화: 해상도 768 (학습 분포 일치) + 품질 prompt 적층 + texture-targeted negative.

photorealistic detailed skin, natural head movement, soft natural blinks, gentle expression NEG: plastic skin, waxy texture, glossy oily sheen, frozen stiff face

관찰: 텍스처는 좋아짐. 입은 그대로.

③ EchoMimic v3 · Config C
768 · 3.24s · wall 632s
비용: $0.086
구조적 확정 5개 독립 변수 적층 — 전부 무효.
실험 조건 — 5개 변수 동시 적층

1. Reference: 3/4각 hero → 정면 test_2.png
2. Audio: +500ms 무음 anchor + EBU R128 loudnorm
3. Text CFG: 6.0 → 5.0
4. Audio CFG (립싱크 다이얼): 3.0 → 3.5
5. Drift-targeted negative ("jaw misaligned, lips out of sync"...)

결론: 5개 직교 변수 모두 동일 실패 → 튜닝 문제 아닌 학습 데이터 문제 (chinese-wav2vec2에 한국어 음운 표상 없음).

④ Sprite-Puppet 리그
1080p60 · 다람찌 · LoRA atlas
비용: ≈ $10 (LoRA 학습 포함)
실패 표정 sprite 간 캐릭터 드리프트 — "다른 다람찌가 나타난다".
실험 조건 — 또 다른 OSS 접근

스택: Qwen-Image + 자체 LoRA (rank 64, 1500 steps) → 10개 표정 sprite 생성 → Playwright + three.js 합성.

실패 원인: 각 표정 sprite를 독립 inference로 생성 → 라이팅·헤드셋 위치·털 색조가 frame 간에 미세하게 다름. 300ms crossfade로도 해결 불가.

의미: 두 번째 독립 OSS 접근도 실패 — OSS 한계는 한 모델 이슈가 아닌 카테고리 전반의 현실.

OSS 종합 두 가지 독립 접근 (audio-driven · sprite-rig) 모두 실패. 공통 원인은 다르지만 결론은 같음 — OSS만으로는 출시 가능 품질에 도달하지 못합니다.

  • 상품을 직접 들 수 없음 — 모든 상품 인서트는 별도 촬영/합성 결합 필요 → 추가 제작 단계.

설령 단가가 내려가도 완벽한 사람은 만들 수 없습니다.
한계는 가격 아래에도 존재합니다.

06

전환 가설 — Obviously-AI 캐릭터

정공법이 막혔다면, 시장이 이미 검증한 우회로가 있습니다 — 사람을 흉내내지 않는, 명백한 AI 캐릭터.

AI인 척 안 하고 AI라고 인정하면
모든 제약이 풀린다.

6.1 핵심 개념

품질 기준을 '사람과 구분 불가'에서 '캐릭터로 일관되고 매력적'으로 의도적으로 내립니다. 이 한 줄의 결정이 품질·비용·기술 한계 세 가지를 동시에 풉니다.

6.2 근거 1 — 언캐니 밸리 (Uncanny Valley)

마사히로 모리 (1970)의 고전적 발견 — 대상이 사람과 유사하지만 '약간' 어긋날 때 시청자 거부감이 최대치로 치솟습니다.

  • 사람에서 멀수록 거부감 낮음 (캐릭터·만화·로봇)
  • 사람에 가까울수록 급격히 떨어짐 → 깊은 골짜기 (Valley)
  • 완전한 사람에 도달하면 다시 호감 회복

현 AI 토킹헤드는 정확히 골짜기 안에 있습니다. 골짜기를 넘는 것보다, 골짜기 앞 — 캐릭터 영역에 머무는 게 훨씬 싸고 안전합니다.

6.3 근거 2 — 한국 시장 선례

한국 시청자는 이미 명백한 AI/캐릭터 콘텐츠에 충분한 시청 시간과 지불 의향을 보이고 있습니다.

① 김햄찌

햄스터 의인화 캐릭터 IP. 정서 불안 콘텐츠로 십수만 팔로워 규모. 핵심 — 캐릭터임이 명백한데도 시청자가 정서적 친밀감을 형성.

참고: 인스타그램 / 유튜브 공개 채널. 구체 view-count는 CEO 검토 시 별첨 확인.

② "과일 사건"

의도적 AI/캐릭터 컨텐츠가 TikTok·인스타에서 수백만 조회. 분명한 합성·과장 연출이 거부감 대신 흥미로 작동.

참고: 별첨 — 실제 view count·engagement 데이터 첨부 예정.

③ 네이버 쇼핑라이브 VTuber

Live2D 캐릭터 호스트가 실제 커머스 매출을 발생시키며 정상 운영 중. 카테고리는 이미 시장 검증 통과.

참고: shoppinglive.naver.com

To-do CEO 검토 단계에서 김햄찌·과일사건·VTuber 커머스에 대한 구체 view count·engagement·매출 데이터를 1–2일 안에 보강해 첨부합니다. 현재는 정성적 시장 관찰 단계.

6.4 이 결정이 푸는 것들

제약실사 호스트에서캐릭터 호스트에서
입 모양 정밀도 최고가 API조차 미달 캐릭터 입은 단순 → 어긋남이 캐릭터다움으로 흡수
제스처 다양성 '말하는 사람' 모드만 단순화된 캐릭터 모션이 더 자연스럽게 보임
단가 $414–$1,080 / hr OSS 시도 가능 → $1–10 / hr 가능성
시청 거부감 언캐니 밸리 안 밸리 진입 전 — 거부감 자체가 발생하지 않음

6.5 솔직한 캐비어트 — sprite-puppet 1차 시도는 실패

선례 Phase 0 v2에서 '명백한 AI 캐릭터' 가설을 sprite-puppet 형태로 시험했고 실패했습니다 (§4.1 영상 ④). 실패 원인은 '캐릭터 가설 자체'가 아니라 '독립 프레임 합성 방식' — 각 표정 sprite가 독립 inference로 생성되어 캐릭터 정체성이 frame 간에 흔들렸습니다. 동영상 모델로 한 번에 생성하면 이 문제는 사라집니다.

07

제안 파이프라인 (전환 가정)

전환이 승인될 경우의 v1 아키텍처 청사진. 본격 PRD는 승인 직후 작성.

7.1 사전 렌더링 + 실시간 렌더링 하이브리드

방송 시간의 ≈ 80%는 사전 작성 가능한 멘트 (상품 설명·가격·소구점), ≈ 20%만 실시간 댓글 반응. 80%를 사전 렌더링 캐싱하면 시간당 비용이 한 자릿수로 떨어집니다.

사전 렌더링이 비싼 (고품질) 모델을 흡수하고, 실시간은 빠른 (저렴한) OSS로 메꿈 → 평균 단가↓.

7.2 비트 구조 (Script Beats)

스크립트를 짧은 비트(beat)로 분해하면 사전/실시간 혼합과 셀러 편집이 모두 쉬워집니다.

[01] 인사 / 오프닝 [사전]
[02] 댓글 응답 슬롯 [실시간]
[03] 상품 1 · 특징 A [사전]
[04] 상품 1 · 사이즈/색상 Q&A [실시간]
[05] 상품 1 · 특징 B / 사용 시나리오 [사전]
[06] 상품 1 · 가격 강조 / CTA [사전]
[07] 친밀 잡담 / 댓글 호명 [실시간]
[08] 상품 2 · 특징 A [사전]
...
[N] 마무리 / 다음 방송 예고 [사전]

7.3 스크립트 생성 흐름

  1. 셀러 입력 — 상품 URL 또는 ShopLive 연동으로 카탈로그 수신.
  2. AI 상품 선별 — 상품이 많을 경우 LLM이 N개 이하로 자동 선별, 셀러가 최종 수정.
  3. 스크립트 작성 — Claude API 등 long-form LLM이 비트 단위 분해 + 멘트 작성.
  4. 셀러 Beat Editor 검토 — 비트별 멘트 확인 / 편집 / 순서 변경 (UI는 §7.5).
  5. 승인 → 사전 렌더링 시작 — 사전 비트들이 큐에 들어가 비싼 모델로 일괄 처리.
  6. 방송 중 — 사전 비트 재생, 실시간 비트는 댓글 흐름을 보며 동적으로 삽입.

7.4 실시간 인터랙션 — 세 카테고리

"실시간 댓글 반응"이라 부르는 것은 사실 성격이 다른 세 가지의 합. 각각 처리 방식이 다릅니다.

유형예시처리 방식
① 자주 묻는 제품 질문 "이 옷 입은 모델 키 몇이에요?" RAG에서 즉시 검색 → LLM이 캐릭터 톤으로 응답 → 실시간 렌더. 캐싱 대상.
② 개별 상황 질문 "165에 55사이즈인데 맞을까요?" RAG + LLM → 셀러 승인 대기 큐로 → 셀러 1-click 승인 후 실시간 렌더. (불확실 답변 보호)
③ 친밀 잡담 "다람찌 오늘 너무 귀여워요" LLM 즉시 캐릭터 응답 — 정확도 위험 없음. 풀 자동.

7.5 셀러 운영 UI (Beat Editor)

핵심 원칙 — 모든 비트는 셀러가 보고 / 편집할 수 있어야 한다.

  1. 모든 질문·답변이 비트 단위로 시각화 — 어떤 질문에 어떤 답이 가는지 한눈에 보임.
  2. 불확실한 답변에 명시 표시 — "좋은 질문이지만 정보 부족" 상태 → 셀러가 직접 채워 LLM 정제 후 비트화.
  3. 비트 순서를 드래그 조정해서 방송 흐름 제어 가능.
  4. 모든 답변을 편집하거나 "답변 안 함" 처리 가능.

이 UX는 일반 SaaS 작업 — 기술 위험 낮음, 차별화 포인트는 셀러 워크플로 디자인.

사전 렌더링과 실시간 렌더링을 갈라놓는 순간, 가장 비싼 도구는 가장 적은 곳에만 쓰이고, 가장 빠른 도구는 가장 많은 곳에 쓰인다. 경제성은 비트 사이의 경계선에서 만들어진다.
